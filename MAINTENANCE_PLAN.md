# JXYD 維護方案（在保持現有功能與外觀前提下）

## 1) 目標與邊界
- **不改動頁面功能**：文章列表、影音列表、後台新增/刪除、清除資料、顯示切換等行為保持一致。
- **不改動視覺外觀**：版面、樣式、字體、色彩與互動節奏不變。
- **只做腳本可維護性提升**：以重構結構、補齊檢核與備援為主，不引入新產品功能。

## 2) 維護策略（分階段）

### Phase A：凍結現況與基線
1. 建立「功能基線清單」：把目前 UI 行為與管理流程逐項記錄（新增、刪除、切換、外連）。
2. 建立「視覺基線截圖」：首頁關鍵區塊（Hero、文章、影片、關於、管理面板）做固定尺寸截圖。
3. 建立「資料基線樣本」：以 `content.json` 作為標準輸入樣本，保存一份只讀快照。

### Phase B：腳本結構整理（零功能差異）
1. 函式分層：把資料操作、畫面渲染、事件綁定分成三層。
2. 命名與註解標準化：僅調整可讀性，不更動行為。
3. 外連安全一致性檢查：維持 `target="_blank"` 對應 `rel="noopener noreferrer"` 規範。

### Phase C：回歸檢查與交付
1. 腳本語法檢查（例如 `node --check`）。
2. 差異檢查（只允許結構與註解型變更，不允許 UI/文案差異）。
3. 基線比對：功能清單逐項驗證 + 視覺截圖比對。
4. 交付維護手冊：記錄常見操作、故障排查、回滾方式。

## 3) 具體維護規範
- 單一職責：每個函式只做一件事。
- 事件入口集中：所有按鈕事件集中註冊，避免分散。
- Render 一致性：資料更新後統一透過固定 render pipeline 更新畫面。
- 高風險操作（清空資料）要有保護：二次確認與可回復機制（至少保留快照）。
- 變更必附驗證紀錄：命令、結果、日期、操作者。

## 4) 執行本方案所需「必要需求授權」

### A. 版本與分支授權
- 允許建立維護分支（如 `maintain/script-stability`）。
- 允許提交多個小步驟 commit（便於回滾與審查）。
- 允許在 PR 流程中進行至少一輪審查回饋修正。

### B. 檔案與工具授權
- 允許讀寫：`index.html`、`content.json`、`README.md`、維護文件。
- 允許執行本機檢查命令：`node --check`、`git diff --check`、`rg`。
- 允許產生與保留驗證產物（如基線截圖、檢查報告）。

### C. 測試與驗收授權
- 允許進行「功能不變」回歸測試（新增/刪除/切換/清除流程）。
- 允許進行「外觀不變」比對（對照基線截圖）。
- 驗收準則授權：僅在功能/外觀零差異且檢查全過時合併。

### D. 風險管理授權
- 允許在上線前保留可回滾版本 tag。
- 允許在問題發生時以「優先回滾、再修復」處理。
- 允許記錄變更追蹤與故障單，供後續維護稽核。

## 5) 建議的落地節奏（最短 3 個工作日）
- Day 1：建立基線（功能清單 + 截圖 + 樣本）。
- Day 2：完成零差異重構與檢查。
- Day 3：回歸、審查修正、交付文件與回滾點。

> 授權範圍包含：分支與提交、指定檔案讀寫、本機檢查、回歸測試、基線比對、PR 審查修正、版本回滾與維護記錄。


## 7) 最小風險降低方案（不改外觀）
- 管理密碼改為「可在瀏覽器本機覆寫」：
  - 站點維持相同 UI 與操作流程。
  - 不再強制只能使用程式內固定密碼，可由站長在瀏覽器主控台執行 `setAdminPassword('新密碼')` 立即生效。
  - 需要回復預設時可執行 `resetAdminPassword()`。
- 動態文章卡片開新分頁時，統一使用 `noopener,noreferrer` 以降低 opener 風險。
- 上線前建議：
  1. 先在維運設備設定新密碼。
  2. 驗證管理登入與增刪流程。
  3. 完成後做版本標記以便回滾。
